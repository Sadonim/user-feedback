# Phase 1 Foundation — Design Handoff

## Overview

This document covers the complete design specification for Phase 1 of the user-feedback system. It includes the Prisma schema, Zod validators, API utility helpers, REST endpoint contracts, UI component trees, file structure, and implementation order. All downstream agents (implementer, tdd-guide, code-reviewer) should treat this as the authoritative spec.

## 1. Prisma Schema

### File: `prisma/schema.prisma`

```prisma
generator client {
  provider = "prisma-client-js"
}

datasource db {
  provider  = "postgresql"
  url       = env("DATABASE_URL")
  directUrl = env("DIRECT_URL")
}

// ─── Enums ───────────────────────────────────────────────

enum FeedbackType {
  BUG
  FEATURE
  GENERAL
}

enum TicketStatus {
  OPEN
  IN_PROGRESS
  RESOLVED
  CLOSED
}

enum UserRole {
  ADMIN
  MANAGER
}

// Phase 5 enum — defined now so migrations are additive only
enum Priority {
  LOW
  MEDIUM
  HIGH
  CRITICAL
}

// ─── Models ──────────────────────────────────────────────

model Feedback {
  id          String       @id @default(cuid())
  type        FeedbackType
  status      TicketStatus @default(OPEN)
  title       String       @db.VarChar(200)
  description String       @db.Text
  nickname    String?      @db.VarChar(100)
  email       String?      @db.VarChar(320)
  userId      String?
  trackingId  String       @unique
  // Phase 5 fields — nullable so they can be added without data migration pain
  priority    Priority?
  assigneeId  String?

  createdAt DateTime @default(now())
  updatedAt DateTime @updatedAt

  // Relations
  user          User?          @relation(fields: [userId], references: [id], onDelete: SetNull)
  assignee      User?          @relation("AssignedFeedback", fields: [assigneeId], references: [id], onDelete: SetNull)
  statusHistory StatusHistory[]

  @@index([status])
  @@index([type])
  @@index([createdAt])
  @@index([userId])
}

model User {
  id           String   @id @default(cuid())
  email        String   @unique @db.VarChar(320)
  username     String   @unique @db.VarChar(50)
  passwordHash String
  role         UserRole @default(ADMIN)
  createdAt    DateTime @default(now())

  // Relations
  feedbacks         Feedback[]
  assignedFeedbacks Feedback[]      @relation("AssignedFeedback")
  statusChanges     StatusHistory[]
}

model StatusHistory {
  id          String       @id @default(cuid())
  feedbackId  String
  fromStatus  TicketStatus?
  toStatus    TicketStatus
  changedById String?
  note        String?      @db.Text
  createdAt   DateTime     @default(now())

  // Relations
  feedback  Feedback @relation(fields: [feedbackId], references: [id], onDelete: Cascade)
  changedBy User?    @relation(fields: [changedById], references: [id], onDelete: SetNull)

  @@index([feedbackId])
  @@index([createdAt])
}
```

### Schema Notes

- `trackingId` is set by the API layer (not auto-generated by Prisma) using the pattern `FB-<nanoid(8)>`.
- `userId` and `assigneeId` are both nullable to support anonymous submissions and pre-Phase-5 operation.
- All `onDelete` behaviors are explicit: `Cascade` for StatusHistory (child records die with parent), `SetNull` for nullable foreign keys.
- The `Priority` and `assigneeId` fields are included now as nullable to avoid breaking schema migrations in Phase 5.

## 3. Zod Validators

### File: `src/lib/validators/feedback.ts`

```typescript
import { z } from 'zod';
import { FeedbackType } from '@prisma/client';

// ─── Field rules ─────────────────────────────────────────

const TITLE_MIN = 5;
const TITLE_MAX = 200;
const DESCRIPTION_MIN = 10;
const DESCRIPTION_MAX = 5000;
const NICKNAME_MAX = 100;

// ─── Schemas ─────────────────────────────────────────────

export const createFeedbackSchema = z.object({
  type: z.nativeEnum(FeedbackType, {
    errorMap: () => ({ message: 'type must be BUG, FEATURE, or GENERAL' }),
  }),
  title: z
    .string({ required_error: 'title is required' })
    .trim()
    .min(TITLE_MIN, `title must be at least ${TITLE_MIN} characters`)
    .max(TITLE_MAX, `title must be at most ${TITLE_MAX} characters`),
  description: z
    .string({ required_error: 'description is required' })
    .trim()
    .min(DESCRIPTION_MIN, `description must be at least ${DESCRIPTION_MIN} characters`)
    .max(DESCRIPTION_MAX, `description must be at most ${DESCRIPTION_MAX} characters`),
  nickname: z
    .string()
    .trim()
    .min(1, 'nickname cannot be empty if provided')
    .max(NICKNAME_MAX, `nickname must be at most ${NICKNAME_MAX} characters`)
    .optional(),
  email: z
    .string()
    .trim()
    .email('email must be a valid email address')
    .optional()
    .or(z.literal('')),
});

// Inferred TypeScript type from the schema
export type CreateFeedbackSchema = z.infer<typeof createFeedbackSchema>;

// ─── trackingId path param ────────────────────────────────

export const trackingIdParamSchema = z.object({
  trackingId: z
    .string()
    .regex(/^FB-[A-Za-z0-9]{8}$/, 'invalid tracking ID format'),
});

export type TrackingIdParam = z.infer<typeof trackingIdParamSchema>;
```

### File: `src/lib/api/error-handler.ts`

```typescript
import { ZodError } from 'zod';
import { Prisma } from '@prisma/client';
import { badRequest, internalError, notFound } from './response';
import type { NextResponse } from 'next/server';
import type { ApiResponse } from '@/types/feedback';

/**
 * Central error handler for API route handlers.
 * Converts known error types into structured API responses.
 * Never leaks stack traces or internal details to the client.
 */
export function handleApiError(
  error: unknown,
): NextResponse<ApiResponse<never>> {
  // Validation error from Zod
  if (error instanceof ZodError) {
    const message = error.errors
      .map((e) => `${e.path.join('.')}: ${e.message}`)
      .join('; ');
    return badRequest(message);
  }

  // Prisma known errors
  if (error instanceof Prisma.PrismaClientKnownRequestError) {
    if (error.code === 'P2025') {
      return notFound('Record not found');
    }
    if (error.code === 'P2002') {
      return badRequest('A record with this value already exists');
    }
    // Log internally but return generic message
    console.error('[Prisma Error]', error.code, error.message);
    return internalError();
  }

  // Generic Error
  if (error instanceof Error) {
    console.error('[API Error]', error.message, error.stack);
    return internalError();
  }

  console.error('[Unknown Error]', error);
  return internalError();
}
```

```typescript
import { customAlphabet } from 'nanoid';

// URL-safe alphanumeric, 8 chars → 218 trillion combinations
const nanoid = customAlphabet('ABCDEFGHIJKLMNOPQRSTUVWXYZabcdefghijklmnopqrstuvwxyz0123456789', 8);

/**
 * Generates a public-facing tracking ID in the format FB-XXXXXXXX.
 * Called once per feedback submission. The generated ID is stored in
 * the trackingId column and returned to the submitter immediately.
 */
export function generateTrackingId(): string {
  return `FB-${nanoid()}`;
}
```

## 6. POST /api/v1/feedback — Endpoint Design

### Route File: `src/app/api/v1/feedback/route.ts`

#### Request

| Field | Type | Required | Rules |
|---|---|---|---|
| `type` | `"BUG" \| "FEATURE" \| "GENERAL"` | Yes | Must be valid enum value |
| `title` | `string` | Yes | 5–200 chars, trimmed |
| `description` | `string` | Yes | 10–5000 chars, trimmed |
| `nickname` | `string` | No | 1–100 chars if present |
| `email` | `string` | No | Valid email format if present |

**Content-Type:** `application/json`

**Example Request Body:**
```json
{
  "type": "BUG",
  "title": "Submit button not responding",
  "description": "When I click the submit button on the form, nothing happens. Tested on Chrome 120.",
  "nickname": "alice",
  "email": "alice@example.com"
}
```

#### Response — 201 Created

```json
{
  "success": true,
  "data": {
    "trackingId": "FB-aB3kLm9Z",
    "status": "OPEN",
    "type": "BUG",
    "title": "Submit button not responding",
    "createdAt": "2026-03-15T09:00:00.000Z"
  },
  "error": null
}
```

#### Error Cases

| Scenario | HTTP Status | `error` field |
|---|---|---|
| Missing required field (`type`, `title`, or `description`) | 400 | `"body.type: type must be BUG, FEATURE, or GENERAL"` |
| `title` too short / too long | 400 | `"body.title: title must be at least 5 characters"` |
| `description` too short / too long | 400 | `"body.description: description must be at least 10 characters"` |
| `email` is present but malformed | 400 | `"body.email: email must be a valid email address"` |
| Invalid JSON body | 400 | `"Bad request"` |
| Database write failure | 500 | `"Internal server error"` |

#### Handler Logic (Pseudocode)

```
1. Parse request.json() — on JSON parse failure → badRequest()
2. Validate body with createFeedbackSchema.parse(body) — on ZodError → handleApiError()
3. Generate trackingId via generateTrackingId()
4. prisma.feedback.create({ data: { ...validatedBody, trackingId, status: OPEN } })
5. Create initial StatusHistory entry: { feedbackId, fromStatus: null, toStatus: OPEN }
6. Return createdResponse({ trackingId, status, type, title, createdAt })
```

#### TypeScript Signature

```typescript

// Only POST is allowed on this route — return 405 for others via Next.js notFound or explicit handler
```

## 7. GET /api/v1/feedback/:trackingId — Endpoint Design

### Route File: `src/app/api/v1/feedback/[trackingId]/route.ts`

#### Request

| Parameter | Location | Type | Rules |
|---|---|---|---|
| `trackingId` | URL path segment | `string` | Must match `^FB-[A-Za-z0-9]{8}$` |

No request body.

**Example:** `GET /api/v1/feedback/FB-aB3kLm9Z`

#### Response — 200 OK

```json
{
  "success": true,
  "data": {
    "trackingId": "FB-aB3kLm9Z",
    "type": "BUG",
    "status": "IN_PROGRESS",
    "title": "Submit button not responding",
    "description": "When I click the submit button on the form, nothing happens.",
    "nickname": "alice",
    "createdAt": "2026-03-15T09:00:00.000Z",
    "updatedAt": "2026-03-15T11:30:00.000Z",
    "statusHistory": [
      {
        "fromStatus": null,
        "toStatus": "OPEN",
        "note": null,
        "createdAt": "2026-03-15T09:00:00.000Z"
      },
      {
        "fromStatus": "OPEN",
        "toStatus": "IN_PROGRESS",
        "note": "Under investigation",
        "createdAt": "2026-03-15T11:30:00.000Z"
      }
    ]
  },
  "error": null
}
```

**Privacy Note:** The response intentionally omits `email`, `userId`, `assigneeId` — these are internal fields.

#### Error Cases

| Scenario | HTTP Status | `error` field |
|---|---|---|
| `trackingId` fails regex | 400 | `"trackingId: invalid tracking ID format"` |
| No record found for `trackingId` | 404 | `"Feedback not found"` |
| Database read failure | 500 | `"Internal server error"` |

#### Handler Logic (Pseudocode)

```
1. Validate params.trackingId with trackingIdParamSchema — on ZodError → handleApiError()
2. prisma.feedback.findUnique({
     where: { trackingId },
     include: { statusHistory: { orderBy: { createdAt: 'asc' } } }
   })
3. If null → notFound("Feedback not found")
4. Map to FeedbackPublicView (strip email, userId, etc.)
5. Return successResponse(publicView)
```

## 8. Feedback Submission Form — Component Tree

## 14. Acceptance Criteria

Phase 1 is complete when ALL of the following pass:

- [ ] `prisma migrate dev` applies cleanly with no errors
- [ ] `POST /api/v1/feedback` returns 201 with `trackingId` for valid input
- [ ] `POST /api/v1/feedback` returns 400 with a clear `error` message for each invalid field
- [ ] `GET /api/v1/feedback/:trackingId` returns 200 with full `FeedbackPublicView` for a valid ID
- [ ] `GET /api/v1/feedback/:trackingId` returns 404 for an unknown tracking ID
- [ ] `GET /api/v1/feedback/invalid-format` returns 400 (schema validation)
- [ ] `/submit` page renders the 3-step form and successfully submits to the API
- [ ] After submission, `trackingId` is shown to the user on the success screen
- [ ] `/track` page accepts a tracking ID and displays the ticket status + history
- [ ] `/track?id=FB-XXXXXXXX` pre-fills and auto-triggers the lookup
- [ ] No TypeScript errors with `strict: true`
- [ ] No internal error details (stack traces, DB errors) leak in API responses

## 15. Out of Scope for Phase 1

The following are explicitly NOT part of Phase 1 and should not be implemented:

- Admin authentication (NextAuth) — Phase 2
- Ticket management endpoints (`GET/PATCH/DELETE /api/v1/tickets`) — Phase 2
- Rate limiting (Upstash Redis) — Phase 4
- Email notifications — Phase 4
- Priority levels or assignee fields (columns defined in schema but not exposed via API) — Phase 5
- Embeddable widget — Phase 3
- Seed data for admin accounts — Phase 2
